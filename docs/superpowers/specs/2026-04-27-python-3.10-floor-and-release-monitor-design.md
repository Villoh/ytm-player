# v1.7.1 — Python 3.10 floor + release monitor

**Date:** 2026-04-27
**Target release:** v1.7.1 (patch)
**Theme:** Broader compatibility + smarter monitoring

## Goal

Lower ytm-player's Python floor from 3.12 to 3.10 so Ubuntu 22.04 LTS users can install via stock `apt python3` + `pip install ytm-player` without extra steps. Add a small automated watcher so we hear about new Python stable releases without manual polling.

## Background

- `pyproject.toml` currently declares `requires-python = ">=3.12"`. This excludes Ubuntu 22.04's default `python3` (3.10).
- Ubuntu 22.04 LTS is supported until April 2027 (standard) / 2032 (Pro). It's the most-deployed LTS distro right now.
- Codebase audit confirms zero use of PEP 695 type aliases, generic class/function syntax, or `@override` — i.e. no 3.12-only syntax exists in `src/`. Lowering the floor is config-only, no code refactor.
- All declared deps (textual, ytmusicapi, pyobjc, etc.) support 3.10+.
- Python 3.10 reaches CPython EOL October 2026 (~6 months from this spec). Worth honest disclosure in the changelog.
- Python core releases happen yearly in October; 3.14.4 is current (released 2026-04-07). Without a watcher we tend to find out months late.

## Scope

### In scope
1. Lower Python floor to 3.10 across config, CI, docs.
2. Update CI matrix to floor + ceiling: `[3.10, 3.14]`.
3. Add a monthly cron action that opens an issue when a newer Python stable exists than what's in our matrix ceiling.
4. Local + CI verification on 3.10.
5. Changelog + version bump + ship.

### Out of scope (separate tasks)
- Dependabot grouping config audit (deferred — not a flood yet).
- mpv release watcher (mpv is user-installed via distro, we don't bundle it).
- Ubuntu LTS release watcher (low value — releases are well-publicized 2-yearly events).
- `flake.nix` Python pin (stays at python312; Nix users can pick whatever Python the flake supplies).
- AUR PKGBUILD (uses unversioned `python` dep — already works on whatever Python Arch ships).
- Adding ruff `UP` rule group (would explode diff with unrelated cleanup; separate concern).

### Watcher design (`check-python-versions.yml`)

```yaml
name: Check Python versions

on:
  schedule:
    - cron: "0 9 1 * *"  # 09:00 UTC on the 1st of each month
  workflow_dispatch:      # manual trigger for testing

permissions:
  issues: write           # to open issues
  contents: read          # to read the workflow file (matrix discovery)

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
      - name: Discover current matrix ceiling
        id: matrix
        run: |
          # Extract the highest python-version from ci.yml matrix.
          ceiling=$(grep -oP 'python-version:\s*\["[^"]+",\s*"\K[^"]+' .github/workflows/ci.yml | head -1)
          echo "ceiling=$ceiling" >> "$GITHUB_OUTPUT"
      - name: Check upstream
        id: upstream
        run: |
          # endoflife.date is a stable, well-maintained public API.
          latest=$(curl -fsS https://endoflife.date/api/python.json | jq -r '.[0].latest' | cut -d. -f1-2)
          echo "latest=$latest" >> "$GITHUB_OUTPUT"
      - name: Open issue if outdated
        if: steps.upstream.outputs.latest != steps.matrix.outputs.ceiling
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          # Skip if an issue with this title already exists.
          title="Bump CI Python ceiling to ${{ steps.upstream.outputs.latest }}"
          existing=$(gh issue list --search "$title in:title is:open" --json number --jq '. | length')
          if [ "$existing" -eq 0 ]; then
            gh issue create \
              --title "$title" \
              --label maintenance \
              --body "Python ${{ steps.upstream.outputs.latest }} is the current stable release (per endoflife.date). CI matrix ceiling is ${{ steps.matrix.outputs.ceiling }}. Time to bump."
          fi
```

Design notes:
- **endoflife.date** — picked over python.org scraping because it returns clean JSON. Source is well-maintained, MIT-licensed, used by other tools.
- **Idempotent**: existing-issue check prevents the workflow from creating a new issue every month if the ceiling stays behind.
- **Manual trigger included** (`workflow_dispatch`) so we can verify it works at landing time without waiting for the 1st of next month.
- **Permissions scoped minimally**: `issues: write`, `contents: read`. No other scopes.
- **Self-resolves**: when ceiling is bumped to match upstream, future runs no-op silently.

### Changelog entry shape (with 3.10 EOL disclosure)

```markdown
### v1.7.1 (2026-04-27)

A compatibility patch that lowers the supported Python floor.

**Project**

- Python floor lowered from 3.12 to 3.10. Ubuntu 22.04 LTS users can now `pip install ytm-player` against the system `python3` without installing a newer Python first. Verified by CI matrix `[3.10, 3.14]`.
  - Note on Python 3.10 lifecycle: CPython 3.10 reaches end-of-life October 2026. Ubuntu 22.04 keeps shipping 3.10 until April 2027 (standard support) or 2032 (Pro), so 22.04 users stay covered well past CPython's EOL. We'll bump the floor when usage data shows nobody on 3.10.
- CI matrix shifted from `[3.12, 3.13]` to `[3.10, 3.14]` — testing the supported floor + the latest stable. Same 6 jobs as before (3 OSes × 2 Pythons), better-targeted coverage.
- New monthly workflow `check-python-versions.yml` opens a maintenance issue when CPython releases a new stable major.minor version newer than our CI matrix ceiling. Idempotent (won't reopen if an issue is already open).
- Pyright + ruff configured to type-check and lint against `py310` so accidentally-introduced 3.11+ syntax fails locally and in CI.
- Classifiers updated: now lists 3.10, 3.11, 3.12, 3.13, 3.14.
```

## Risks

- **Low risk overall** — additive change, no breaking surface.
- **One real risk**: a dep we don't notice has tightened its own floor to 3.11 since we last looked. Mitigation: CI runs `pip install -e ".[dev]"` on 3.10. If a transitive dep has dropped 3.10 support, CI will fail at install time. We'll either pin that dep below the breaking version or accept that the floor stays at the lowest version every dep supports.
- **Watcher false positives**: endoflife.date might briefly list a release candidate as "latest" right after a new minor drops. Mitigation: the workflow only triggers on the major.minor cut (`cut -d. -f1-2`), so 3.15.0rc1 → "3.15" matches a real upcoming stable. If this is wrong, we close the issue and move on. Acceptable.

## Acceptance criteria

- `pip install ytm-player` works on Ubuntu 22.04 with no Python upgrade.
- CI matrix shows `[3.10, 3.14]` × `[ubuntu, macos, windows]` = 6 green jobs.
- New cron workflow exists and runs cleanly on manual trigger.
- Changelog entry mentions both the floor change and the 3.10 EOL caveat.
- v1.7.1 published to PyPI, AUR, GitHub Releases.
